# 投稿指南

想把自己的作品（或你欣賞的台灣開發者作品）加進清單？發 Pull Request 或 Issue 都可以。

## 條目格式

完整格式（城市、GitHub、部落格、更多介紹皆為選填）：

```
#### 開發者名字（城市） - [GitHub](連結), [部落格](連結)
* :white_check_mark: [產品名稱](網址)：一句話介紹 - [更多介紹](連結)
```

最精簡的版本（只保留必填項）：

```
#### 開發者名字
* :white_check_mark: [產品名稱](網址)：一句話介紹
```

狀態符號有三種：:clock8: 開發中、:white_check_mark: 已上線、:x: 已關閉或缺乏維護。
已關閉的專案，建議附上「更多介紹」連結，聊聊關閉的原因與學到的事。

## 一句話介紹怎麼寫

讀者掃過每個條目的時間大概只有兩秒，所以：

1. **開頭先定性**——第一個詞就讓人知道這是什麼（記帳 App、解謎遊戲、命令列工具）
2. **接著給出記憶點**——獨特的切入點，或具體的數字（下載量、支援平台、星數）
3. **沒有明顯記憶點時，寫使用門檻**——免費、無需註冊、資料留在本地，這些都有價值

以清單中的真實條目為例：

- 「公車動態查詢 App，整合全台公車、捷運、臺鐵、YouBike 與高鐵訂票」——先定性，再用整合範圍當記憶點
- 「整個世界皆由中文字構成的解謎 RPG，以刪、推、拆、組文字破解謎題」——一句話就能想像玩法
- 「地震預警 App，強震來襲前倒數計時並顯示預估震度，常比國家級警報早 10 秒以上」——具體數字勝過形容詞

反例：「一個很棒的 AI 工具」——看不出是什麼，也記不住。

## 排版慣例

本清單遵循台灣常見的中英混排慣例：

- 中文與英文、數字之間加半形空格：「支援 LaTeX 公式」「300 萬次下載」
- 中文句子使用全形標點：，、。：；（）
- Markdown 語法與狀態符號（如 `:white_check_mark:`）維持半形

## 收錄標準

1. 開發者是台灣人，或以台灣為根據地的小型團隊
2. 獨立開發的作品（個人專案或 side project），而非公司主力產品
3. 主版面收「打開即用」的網站與 App；開源工具請投稿[工程師版面](./pages/README-Programmer-Edition.md)，遊戲請投稿[遊戲版面](./pages/README-Game.md)
